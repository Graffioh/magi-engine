RoPE — turn the arrow, don't add to it

You already know one way to tell a model where a token sits: add a position vector to its embedding — x = token + position. RoPE does it differently: instead of adding something on, it rotates the query & key vectors by an angle that grows with position. Four scenes build the whole idea from that one swap — add → turn — starting exactly where you already are. Drag, slide, and let things spin.

Adding a position vector pushes the tip to a new spot — it drifts off the circle and its length lurches around (watch the purple readout). The token's size, which carries its meaning, gets disturbed. Turning spins the very same arrow along the circle: new heading, length never changes. Position becomes pure angle.

one 2D plane shown — a real head has 32 of them (Scene 3) RoPE turns Q & K inside every attention layer — not once at the input like additive PE

Drawn with a sinusoidal position vector — a fixed-length arrow whose direction turns with position. The point is general, not a rigged contrast: adding any position vector changes the total length, while turning never does.

Turning keeps the vector intact — nice. But the model never looks at one arrow alone. It only ever compares two tokens. So does a turn even survive being compared? →
2 turn both — only the gap survives

When attention compares two tokens, each one becomes a query q (the asker) and a key k (the answerer). The score is just how aligned q and k point — and the green bar below is k's shadow on q, whose length is that alignment (its cosine).

Attention compares two tokens by how aligned their arrows point — the angle between them. Give q the turn for position m and k the turn for position n. Move one and the wedge opens and the score moves. Now slide both forward together: both arrows whirl around, but the angle between them — and the score — stay frozen to the pixel. Attention only ever feels the gap n − m.

attention's score for two tokens is how aligned their arrows are: q · k RoPE turns q by m×θ and k by n×θ before comparing: ( turn q by m×θ ) · ( turn k by n×θ ) a turn on each, then compared, = turning just one by the difference (the rest cancels): = q · ( turn k by (n−m)×θ ) → the absolute positions m, n drop out — only the gap (n−m) is left.

Lengths are pinned to 1 here so the score is the cosine; the real score sums |q||k|·cos over all 32 planes. One representative plane shown — each of the 32 spins at its own speed (next scene).

spun by absolute position → score depends only on relative distance · applied to Q and K only, never V
One turn-speed, though: after a full loop the hand points the same way again, so position 1 and position "1 + one whole turn" would look identical. How does RoPE avoid that collision? →
3 one speed lies — a ladder of speeds doesn't
—

Each clock is one plane, turning at its own speed: fast on the left, slow on the right (θ = pos × 10000−2p/d). A single clock is a liar — it returns to the exact same hand every full turn, so it can't tell two far-apart positions apart. But read the whole row of hands together and every position gets a signature no other position shares: fast hands split nearby tokens, slow hands track long range.

Showing 7 of the 32 planes (the liveliest); planes further right are slower still — nearly still over one sentence. KV-cache offset shifts position to start_pos + s, so a decode step keeps spinning from where the cached context left off.

This engine pairs ADJACENT dims (2p, 2p+1) — the llama.cpp convention. Hover a pair.
plane p — its two dims hover a pair on the strip

freqp = base−2p/d  (base=10000, d=64)
θp = pos × freqp  (pos = start_pos + s)
a′ = a·cos θ − b·sin θ
b′ = a·sin θ + b·cos θ 

freq  = powf(base, -2.f*p/head_dim);  // p in [0, 32)theta = pos * freq;                  // pos = start_pos + sa' = a*cosf(theta) - b*sinf(theta); // dim 2pb' = a*sinf(theta) + b*cosf(theta); // dim 2p+1
applied to Q and K only — never V GQA: n_heads ≥ n_kv_heads, same θ schedule start_pos offsets pos for the KV-cache

Each plane is one 2D rotation on adjacent dims (2p, 2p+1) — the llama.cpp interleaved convention, not half-split. (Both conventions are the same idea — 2D rotations — differing only in which two dims each plane pairs.) Spin Q and K, leave V alone. That's ops::RoPE.

in one line

Don't add a position — turn the arrow. Rotating each token's query & key by its absolute position makes attention depend only on the relative gap — and a geometric ladder of spin-speeds gives every position a unique fingerprint. The whole story compiles down to four lines: a′ = a·cos θ − b·sin θ, b′ = a·sin θ + b·cos θ on dims (2p, 2p+1) of Q and K.
