mmap & model loading — visualization

Loading model weights with mmap instead of read() is the trick that lets llama.cpp "load" a 7B model in milliseconds: the call returns as soon as the kernel has set up page tables, and pages fault in lazily on first touch. Three things to understand: why mmap is fast (no user-space copy, no eager I/O), how the file maps to tensor views (offsets into one shared backing buffer), and how lifetimes are tied together (a shared_ptr keeps the mapping alive as long as any tensor still references it).

mmap returns immediately after building page tables. Pages fault in on first access. Multiple processes mapping the same file share physical pages.

Bonus: mmap'd pages are file-backed, so under memory pressure the kernel drops them (re-faulting from disk later) instead of swapping. read()-loaded buffers are anonymous memory — they go to swap, which is slow and duplicates the file on disk.

demand paging — what "lazy load" actually does

mmap() returned in microseconds, but no bytes are in RAM yet. The kernel only installed page-table entries marking the range as valid. The first time your code reads from a page, the CPU traps to the kernel — a page fault — and the kernel pulls that 4 KB page from disk into the page cache. Subsequent reads to the same page hit DRAM directly. Click a tensor to "run a forward pass" through it and watch its pages fault in.

Why this matters for inference: only weights you actually touch get loaded. If a layer is skipped, its pages never fault in. Under RAM pressure the kernel drops cold file-backed pages — no swap, no copy, just a re-fault from disk if you read them again. There is no cache-management code on your side: the OS does it.

a GGUF file mapped into memory — tensors are views, not copies

One mmap() call maps the entire file. MappedFile holds the (base, size) pair for the whole mapping. Each Tensor stores a raw pointer into the mapping (base + offset) plus a shared_ptr<MappedFile> in its owner_ field. Tensors don't copy bytes — they're typed views over the kernel's page cache.

lifetime — when is the mapping released?

Each Tensor::owner_ holds a shared_ptr referencing the same MappedFile. The mapping is alive while the refcount is greater than zero. Destroy tensors below — when the last one dies, the MappedFile destructor runs and munmap releases the region.

auto mf = std::make_shared<MappedFile>("weights.gguf");

where this lands in the codebase

The current Tensor class holds std::vector<float> data_ — which always heap-allocates and always owns. To support mmap-backed views, data_ needs to become a raw pointer, with owner_ controlling its lifetime. One field, one lifetime model — same shape works for owned tensors, mmap views, and (eventually) GPU buffers.

today — owned only
class Tensor {
    std::vector<float> data_;     // owns a heap buffer
    std::vector<int> shape_, strides_;
    std::shared_ptr<void> owner_;    // dead field
};
target — view-friendly
class Tensor {
    float* data_ = nullptr;       // points anywhere
    std::vector<int> shape_, strides_;
    std::shared_ptr<void> owner_;    // keeps data_ valid
};
PyTorch parallel: their Storage is roughly your MappedFile-or-heap-buffer; Tensor is a typed view (shape, strides, offset, dtype) into a Storage. Multiple tensors share one storage. Slicing is free — child tensors copy the shared_ptr, not the bytes.

references — llama.cpp

The pattern modeled here is lifted almost directly from llama.cpp. Its llama_mmap class wraps one mmap() over the GGUF file; every tensor's data pointer is just mmap base + offset, and the mapping outlives the tensors via shared ownership — same idea, slightly different plumbing.

  • PR #613 — "Use mmap for loading models" (Justine Tunney, March 2023). The original change that took load times from seconds to milliseconds. The PR description is itself a great primer on why mmap wins here.
  • src/llama-mmap.cpp — current implementation of llama_mmap / llama_mlock (with the Windows MapViewOfFile and POSIX mmap branches side by side).
  • src/llama-model-loader.cpp — where the GGUF file is opened, mapped, and tensor metadata is wired up to offsets in the mapping.
  • GGUF format spec — the on-disk layout (header + metadata + tensor table + tensor data) that makes "mmap + offset" sufficient.
