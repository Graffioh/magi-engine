Grouped-Query Attention

Keep every query head, but let groups of them share one key/value head. That single trade is what keeps the KV cache small enough to decode long sequences.

The idea

In plain multi-head attention (MHA) every query head owns a matching key head and value head. During autoregressive decoding we cache K and V for every past token, so the cache grows with the number of heads — and at long context that cache, not the weights, is what dominates memory.

GQA splits the difference: keep all n_heads query heads, but only n_kv_heads key/value heads. Each KV head is shared by a contiguous group of query heads.

group_size = n_heads / n_kv_heads // must divide evenly query head h reads from kv head h / group_size

MHA is GQA with n_kv_heads = n_heads. MQA (multi-query) is GQA with n_kv_heads = 1. magi-engine's golden test config is the MQA extreme — n_heads = 2, n_kv_heads = 1, head_dim = 4 — small enough to print whole tensors while still exercising the shared-KV path; the eventual target model uses true GQA with several KV heads.

Why it matters · KV cache size

Per token, per layer, the KV cache holds 2 · n_kv_heads · head_dim floats — the factor of 2 is K and V. Halving the KV heads halves the cache, with little measured quality loss.

In the magi-engine code

The whole mechanism is one integer division in the attention loop — query head h is mapped to its KV head, then K/V slices are gathered from that shared head:

// src/ops.cpp — ops::attn for (int h = 0; h < n_heads; ++h) { const int kv_head = h / (n_heads / n_kv_heads); // <- the group mapping build_head_slice(q_data_ptr, Q_slice, T, n_heads * head_dim, h, head_dim); build_head_slice(k_data_ptr, K_slice, T, n_kv_heads * head_dim, kv_head, head_dim); ... }

RoPE is applied with the matching head counts too — n_heads for Q, n_kv_heads for K (see rope) — so a query and the KV head it shares are rotated on the same θ schedule.
